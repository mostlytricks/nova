01Mission

local-llmstxt-server is an intranet documentation control plane for coding agents. It helps a human curate the documentation surface agents should trust first — local project docs, internal API notes, and approved external llms.txt sources cached as normalized markdown — and exposes that surface as namespaced manifests, so an agent fetches a focused, relevant slice instead of drowning in (or missing) everything. The calm it creates: an agent doing software work has one stable place to ask "what should I read for this?" and gets back a small, trustworthy answer.

north star: one stable intranet URL that answers "what should an agent read
            first" — small, trusted, namespaced
first wedge: namespace docs-split — turn oversized manifests into focused,
            agent-ready slices (the current active arc)
later shape: a full doc control plane — probe, import, promote, refresh,
            health, and search over own + trusted external docs
Staged, not confused. Starting at docs-split isn't the whole ambition — it's the capability that proves the manifest is actually usable by an agent, which everything else depends on. The control-plane breadth comes after retrieval quality is real.

02Why This First

Oversized manifests are one of the main ways this project quietly fails its mission: an agent can be shown exactly the right docs and still be unable to retrieve and use them well, because the slice is too big to fit usefully in context. Docs-split attacks that failure mode head-on — it forces the hard, real questions (how to slice without mutating originals, how to keep provenance, how an agent picks the right sibling namespace) to the front, where they belong, instead of after a pile of cosmetic features.

03Product Principles

  • Agent-first consumption. Every feature should make it easier for an agent to fetch the right small set of docs — not more docs, the right ones.
  • Human-curated trust. A human decides which external sources become active and why; the server never auto-promotes a source into the trusted surface.
  • Intranet by default. Assume a trusted local network and keep read endpoints simple — but write actions must stay protectable rather than permanently open.
  • Focused namespaces beat giant manifests. Large doc sets are split into smaller, agent-ready slices; a tight namespace is a feature, not a limitation.
  • Simple files are the source of truth. Own docs live as files under data/own/; the SQLite index and cache are reproducible state, safe to rebuild from scratch.
  • Operational clarity over automation magic. Prefer explicit probe / import / promote / refresh / archive / split actions over opaque background behavior an operator can't reason about.

04Architecture Principle

Originals are immutable; everything an agent reads is a view over them. Own docs and cached external content are the source — they keep their provenance (source URL, timestamps, refresh status) and are never mutated to fit a manifest. Namespaces, the master manifest, and every split are non-destructive views built from those originals. A split produces sibling namespaces; it never edits or destroys the namespace it came from. Every feature should extend this seam — produce or refine a view — rather than reach around it to rewrite a source.

  data/own/*  ─┐
  cached ext  ─┼─▶ [ namespace builder ] ─▶ /<ns>/llms.txt  (a view)
  splits      ─┘          (never mutates the source)

  provenance (url · ts · refresh status) travels with every imported/cached doc
  a split emits sibling namespaces — the original namespace is left untouched

06Current Non-Goals

  • Not a general bookmark manager. Scope is docs an agent should trust during software work, not arbitrary link-saving for humans.
  • Not a public documentation crawler. External sources are human-probed and human-promoted, never mass-scraped — trust is a deliberate act.
  • Not a general knowledge base or wiki. It curates and serves an existing doc surface for agents; it doesn't author, host, or become the system of record for knowledge.
  • Not a public-internet, multi-tenant service. Intranet by default; write protection is a planned hardening (see §02 Later), not a pivot to hosted SaaS with accounts.
