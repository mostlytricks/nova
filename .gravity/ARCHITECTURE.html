Architecture

This system turns scattered documentation into curated, validated, locally served llms.txt doc sets that coding agents can read selectively.

01System Model

The core loop is agent-assisted ingestion, local curation, validated doc-set output, then selective agent consumption. A coding agent can help collect and normalize docs, but the server is the stable place where the curated result lives.

source material
  URLs, OpenAPI, existing llms.txt, docs pages, PDFs, decks, pasted specs
        |
        v
producer agent
  routes input, fetches or reads material, asks for missing metadata
        |
        v
data/own/<local-doc>/
  llms.txt manifest + focused markdown entry files
        |
        v
local server
  serves /<namespace>/llms.txt and linked entries
        |
        v
consumer agent
  reads manifest first, fetches only relevant entries, uses them as context
The server is not meant to replace source documentation. It creates an agent-ready control surface over docs humans have chosen to trust.

02Agent Roles

RoleSkillJob
Producer agent docs-import Handles URL inputs that the CLI can probe, fetch, parse, crawl, or render.
Producer agent llms-compose Handles local material the agent reads directly: pptx, PDF, pasted text, local markdown, internal notes, or operator-pasted rendered SPA content.
Protected-source helper cookie-extract Runs human-in-the-loop login, lets the operator authenticate in a browser, then extracts selected session cookies.
Protected-source helper windows-ca-web-fetch Fetches URLs on Windows with custom CA bundles, custom headers, cookies, and cookie jars.
Consumer agent llms-txt-reader Reads a served manifest, selects relevant links by description, fetches only needed entries, and grounds work in those docs.

Producer skills are for collecting and building Local Docs. Consumer agents do not need those producer skills unless they are also importing documentation.

ingest/SPEC.md is the front door for deciding which path to use.

The rule is simple: URLs are probed first; local material is read by the composing agent; missing metadata is asked for, not invented.

04Local Docs Contract

Local Docs are the self-written serving units. Internally each one is a namespace folder under data/own/<namespace>/ and contains one manifest plus linked markdown entries.

data/own/
  billing-api/
    llms.txt
    auth.md
    invoices.md
    payments.md

The manifest is an index for selection. Entry files are the payload. A good local doc set lets another agent decide what to fetch without reading every entry.

# Billing API

> Internal billing service - create invoices, record payments, query balances.

> Source: internal Billing API deck v2. Imported 2026-06-20. Base URL: https://billing.example/v2. Auth: Bearer, see auth.md.

## Reference

- [Authentication](/api/entries/get?name=billing-api/auth.md): Bearer token scheme, scopes, token lifetime
- [Invoices](/api/entries/get?name=billing-api/invoices.md): create, fetch, and void invoices
- [Payments](/api/entries/get?name=billing-api/payments.md): record and reconcile payments against invoices

Validate each local doc set before handing it to another agent:

pnpm docs-import check <namespace>

05Serving Model

Run the app locally or on the intranet:

pnpm dev
URLUse
http://localhost:3000/agent/llms.txtRecommended agent entrypoint. Local master merged with active trusted Imported Docs.
http://localhost:3000/agent/namespacesJSON catalog of Local Docs and their manifest URLs.
http://localhost:3000/agent/sourcesJSON catalog of active Imported Docs and their source-specific manifest URLs.

http://localhost:3000/api/entries/get?name=<namespace>/<file>.mdOne local markdown entry linked from a manifest.
http://localhost:3000/api/content/<hash>Cached normalized external markdown by content hash.

06Consumer Instruction

A worker agent that only needs to use the served docs needs the llms-txt-reader behavior. It should not import, crawl, or compose unless the task explicitly asks it to collect docs. Use a specific /<namespace>/llms.txt Local Docs manifest or /agent/sources/:id/llms.txt Imported Docs manifest when the task already has a narrow domain.

Use http://localhost:3000/agent/llms.txt as the source manifest.
Read the manifest first.
Choose only relevant linked entries by their descriptions.
Fetch those entries and use them as the authoritative docs for this task.
Do not fetch every link unless the task requires broad coverage.

That is the main handoff between this control plane and other coding agents.

07State And Trust

PathMeaning
data/own/Human-curated Local Docs. This is source-of-truth content worth tracking when it is not private.
data/cache/Normalized markdown fetched from remote links. Reproducible cache.
data/index.sqliteImported Docs registry, lifecycle state, metadata, refresh history, and tombstones.
.claude/skills/Project-local Claude Code skill playbooks.
.codex/skills/Project-local Codex skill playbooks.

Imported Docs are human-curated. An import can be probed, added as trial, promoted to active, archived, or removed. Active imports are the ones exposed in merged agent views.

Do not expose this server directly to the public internet. It is designed for trusted local or intranet use. Mutating API routes can be protected with WRITE_TOKEN.
